# Plan: Rune E2E — Hypernetwork as Full Project State Machine

## Core Concept

The hypernetwork carries the ENTIRE project lifecycle state through weight-space. The prompt is just the launch trigger and never changes. Each iteration's output (plan, code, test results, errors) gets encoded into an adapter via hypernetwork, and the NEXT iteration loads that adapter. Context window stays constant because ALL accumulated context lives in adapter weights.

```
Iteration 0 (bootstrap):
  Input to H(): methodology doc (how to plan, execute, code, validate)
  Output: methodology_adapter
  → model now knows HOW to work on projects

Iteration 1 (planning):
  Prompt: "Build a statistics library" (project definition — never changes)
  Adapter: methodology_adapter
  Model output: implementation plan (tasks, architecture, dependencies)
  H(plan trajectory) → plan_adapter

Iteration 2 (coding task 1):
  Prompt: same project definition
  Adapter: plan_adapter (model "knows" the plan through weights)
  Model output: code for first task from the plan
  Execute in sandbox → test results
  H(code + results trajectory) → progress_adapter_v1

Iteration 3 (coding task 2 or fix):
  Prompt: same project definition
  Adapter: progress_adapter_v1 (knows what's done, what failed)
  Model output: next code or fix
  H(trajectory) → progress_adapter_v2

...adapters carry the heavy context. Prompts CAN be adjusted between iterations if useful, but the adapter is the primary context mechanism — the prompt doesn't need to grow to carry prior state.
```

The swarm: multiple agents work on different plan tasks simultaneously, each with their own adapter chain. Evolution merges successful adapters.
