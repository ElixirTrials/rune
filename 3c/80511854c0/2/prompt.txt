## 1. TRAJECTORY SYSTEM

**Location:** `/Users/noahdolevelixir/Code/rune/libs/model-training/src/model_training/trajectory.py`

**How Trajectories Are Recorded:**

- **Function:** `record_trajectory(session_id, steps, outcome, task_description, task_type, adapter_ids)`
- **Format:** JSON file stored at `~/.rune/trajectories/{session_id}.json` (or env var `RUNE_TRAJECTORY_DIR`)
- **Schema:**
  ```python
  {
    "session_id": str,
    "task_description": str,
    "task_type": str,
    "adapter_ids": list[str],
    "outcome": "success" | "exhausted" | None,
    "timestamp": ISO 8601,
    "steps": list[dict]  # Each step contains:
      {
        "generated_code": str,
        "stdout": str,
        "stderr": str,
        "exit_code": int,
        "tests_passed": bool
      }
  }
  ```

**How Trajectories Flow to Training:**

1. **Recording:** Agent loop (`services/rune-agent/src/rune_agent/nodes.py`) calls `record_trajectory()` in `save_trajectory_node()` after loop terminates
2. **Loading:** Training services call `load_trajectory(trajectory_id)` to retrieve the JSON
3. **Formatting for SFT:** `format_for_sft(trajectory)` extracts the final successful step (or latest step if exhausted) and returns 3-message format:
   ```python
   [
     {"role": "system", "content": "You are a Python code generator. Output only code, no explanation."},
     {"role": "user", "content": task_description},
     {"role": "assistant", "content": generated_code}  # from last successful step
   ]
   ```
4. **Empty on Failure:** If outcome != "success" or no step has tests_passed=True, returns empty list `[]`

**Complete Data Flow:**
- Agent loop records per-attempt: code → execution → (pass/fail)
- Complete trajectory persists to disk as JSON
- Training pipeline loads, formats, and consumes the SFT messages
- Adapter metadata stores `session_id` linking back to original trajectory

## 3. HYPERNETWORK CONCEPT

**Location:** `/Users/noahdolevelixir/Code/rune/libs/model-training/src/model_training/hypernetwork.py`

**What is it:** DocToLoraHypernetwork — Perceiver-based instant LoRA adapter generation from token IDs (single forward pass, no gradient descent at inference time)

**Architecture:**

```
Input: token_ids (batch, seq_len) from trajectory text
  ↓
Lightweight token embedding (hypernetwork's own, not base model)
  ↓
Learnable latent array (num_latents=32, latent_dim=256)
  ↓
Cross-attention: latents attend over token embeddings
  ↓
Self-attention stack (4 layers) over latents
  ↓
Linear head: projects to all LoRA weight parameters
  ↓
Output: PEFT state_dict with lora_A, lora_B for each layer/module
```

**Key Parameters:**
- input_dim: vocab size
- num_latents: 32
- latent_dim: 256
- depth: 4 (self-attention layers)
- heads: 8
- rank: 8 (output LoRA rank)
- target_modules: ["q_proj", "v_proj"]
- num_layers: 28 (Qwen2.5-Coder-7B layers)
- hidden_dim: 4096

**Forward Pass Output:**
- Returns dict mapping `"base_model.model.model.layers.{i}.self_attn.{module}.lora_{A|B}.weight"` → tensor
- Each pair (lora_A, lora_B) has shape (rank, hidden_dim) and (hidden_dim, rank)

**Lazy Proxy Pattern:**
- `DocToLoraHypernetwork` is a lazy proxy that only builds the real `nn.Module` class on first instantiation
- Keeps module importable without torch

**Saving:** `save_hypernetwork_adapter(weights, output_dir, base_model_id)` writes:
- `adapter_model.safetensors` (weights)
- `adapter_config.json` (PEFT format config)

**Phase Status:**
- Phase 3: QLoRA direct fine-tuning (implemented, current)
- Phase 4: Hypernetwork forward pass replaces gradient descent (proposed, pending validation)

## 6. ADAPTER FITNESS & EVOLUTION

**Location:** `/Users/noahdolevelixir/Code/rune/scripts/swarm_evolution.py`

**Fitness Calculation:**
```python
phi(a) = alpha * pass_rate(a) + (1 - alpha) * generalization(a)
```
- Default alpha: 0.7 (prefer specialization)
- pass_rate: test-passing rate on held-out tasks
- generalization: cross-task coverage metric (TBD)
- Stored in `fitness_score` column

**Evolution Operations:**

1. **Evaluate:** `evaluate_fitness()` computes and updates fitness
2. **Merge (TIES-Merging):** For each task type with ≥5 adapters:
   - `query_best_for_task(task_type, top_k=3)` gets top 3
   - `ties_merge(state_dicts)` combines them (trim, elect sign, average)
   - Creates new adapter entry with `parent_ids` pointing to sources
3. **Prune:** Archive any adapter with `fitness_score < 0.3`
4. **Promote:** New merged adapters inherit `generation = max(parents) + 1`

**Merge Implementations:**
- **TIES-Merging:** Trim parameters below density threshold, majority-vote sign, average matching values
- **DARE-Merging:** Drop & rescale — randomly drops a fraction, rescales, averages

## 9. CRITICAL DESIGN DECISIONS

1. **Write-Once Adapter Storage:** No adapter is ever overwritten; new versions create separate entries with `parent_ids` tracking lineage
2. **Deferred GPU Imports:** All GPU-dependent code is inside function bodies (INFRA-05) so library is importable on CPU
3. **Single-GPU Time-Sharing:** Training pool coordinates with vLLM via sleep/wake API to share single GPU
4. **Fitness-Driven Evolution:** Empirical evaluation (pass_rate, generalization) before promoting adapters, not assumed safety via orthogonality
5. **Trajectory Persistence:** Complete trajectory (all attempts, all execution output) recorded for both successful and failed tasks
6. **SFT Format Conversion:** Only successful trajectories produce training data; failed trajectories recorded but discarded for training
7. **Lazy Hypernetwork Proxy:** Module imports work in CPU CI; torch only loaded on first instantiation
