# Rune v6.0 — Parallel Swarm Execution Implementation Plan

## Context

The swarm-execution-proposal-v2 collapses Rune's 5-service microservice architecture into a single-process "fat orchestrator" for local hardware. This plan implements the proposal across 5 phases (25–29), each on its own branch with granular commits, where every commit passes ruff, mypy, and pytest.

Key design decisions:
- **No new `libs/swarm` package** — swarm logic lives in `scripts/swarm.py` as described in the proposal. Supporting code (HardwareProbe, sandbox) goes into existing libs (`libs/shared`, `libs/model-training`).
- All GPU/system deps (psutil, pynvml) use INFRA-05 deferred-import pattern with `sys.modules` test injection.
- nsjail is Linux-only; macOS/CI falls back to subprocess (tested explicitly).
- AdapterRecord schema additions use `default=` values so `create_all()` works for new DBs.
